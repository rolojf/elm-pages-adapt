# elm-pages: Deployment & adapters

## Static-only sites: no adapter

If the app uses only `RouteBuilder.single`/`preRender` (and only `single`/`preRender`
ApiRoutes), `elm-pages build` produces a self-contained `dist/` folder. Serve it from
any static host or CDN (Netlify, Vercel, GitHub Pages, nginx...). Nothing else needed.

## Server-rendered routes: adapters

`serverRender` routes (and `serverRender`/`preRenderWithFallback` ApiRoutes) need
server-side execution. An **adapter** glues the elm-pages render function to a
specific host. Configure it in `elm-pages.config.mjs`:

```js
import adapter from "elm-pages/adapter/netlify.js";

export default {
  adapter,
};
```

Netlify is the only built-in adapter. Community adapters for other hosts exist
(github.com/dillonkearns/elm-pages/discussions/378).

## Custom adapter API

An adapter is an async function run after `elm-pages build` (so `dist/` already
exists). It moves/generates whatever the host needs; its return value is ignored.

```js
export default async function run({
  renderFunctionFilePath, // JS file with default-export async render function
  routePatterns,          // route list with .kind and .pathPattern
  apiRoutePatterns,
}) { ... }
```

`route.kind` is `"static"`, `"prerender"`, `"prerender-with-fallback"`, or
`"serverless"` — the last two require routing requests to the render function.

### Invoking the render function

The render function takes a plain request object and returns a response object:

```js
import * as elmPages from "<renderFunctionFilePath>";

const renderResult = await elmPages.render({
  requestTime: Math.round(new Date().getTime()),
  method: req.httpMethod,
  headers: req.headers,
  rawUrl: req.rawUrl,
  body: req.body, // string or null
  multiPartFormData: null,
});
```

`renderResult` has `{ kind, statusCode, headers, body, isBase64Encoded }` where
`kind` is:

- `"html"` — page HTML; serve with `Content-Type: text/html`.
- `"bytes"` — the binary `content.dat` payload for client-side elm-pages navigation;
  serve as `application/octet-stream` (base64-encode if your platform requires it).
- `"api-response"` — an ApiRoute response; honor `isBase64Encoded`.

Each server-rendered page route must also serve `<path>/content.dat` through the
same render function — client-side navigations request it. Example (Netlify-style
redirects generated by an adapter):

```
/users/:id            /.netlify/functions/server-render 200
/users/:id/content.dat /.netlify/functions/server-render 200
```

The full Netlify adapter source is a good reference implementation:
github.com/dillonkearns/elm-pages/blob/master/adapter/netlify.js

## Gotchas

- The dev server (`elm-pages dev`) handles server-rendered routes without an
  adapter; adapters only matter for production deploys.
- Env vars used in BackendTasks must be present in the production environment for
  server-rendered routes (request-time) — not just in CI (build-time).
- Wrap the render call in try/catch and return a 500 page; don't let exceptions
  escape the host's handler.
